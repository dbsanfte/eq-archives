regarding the "stand heal then sit and med again", the program only handles keyboard scancodes and doesn't do a thing for mouse movement (simulating mouse movement is in the works but not a priority). any sequence you can do via keyboard only the macro program can record and replay.

I'm currently modifying the program to record and playback the pauses between keystrokes. makes it easier to macro a "cancel med (ESC key), stand up hotkey, target party member (F2-F6), switch hotkey bank, press hotkey (i.e. heal spell), [DELAY for spell to be cast], switch hotkey bank, press hotkey (i.e. spellbook hotkey)" where you can target a different party member with a different hotkey. you will, however, have to click the med button manually when the macro is done.

I'm also binding joystick buttons/levers and/or axis/rudder controls to various hotkey sequences. Keyboard VXD is complete. 100% working keyboard/mouse macro recording and playback with no problems whatsoever. Joystick to keyboard remapping is working for the BUTTONS but not for the axis (don't ask).

Due to the way EQ processes keystrokes, no buffering is possible. That means that the timing on the macro MUST take into account recycle times and LAG. If you're hit with a long lag packet, you macro will cause all kinds of bad things to happen. Timing is critical in EQ, especially for unattended operations (not recommended).

The macro program as it stands right now is what I'm going to release in about a week. I just want a chance to slap together some documentation, strip out the debug info, and put together a nice long disclaimer.

A solution to the timing problem is in the works but it's not 100% reliable. VFLATD provides access to linear video ram at the vxd level. By putting some restrictions on where consumable can be placed, I can detect when a hotkey is ready to reuse, when a spell is done recycling, and when a consumable item has been used up (i.e. for macroing brewing).

There's still problems that need to be worked out, such as items on the cursor when a skill succeeds (or fails and created a secondary item), the little numeric counts on stacks of items, and the 3 video resolutions that EQ allows. All these need to be addressed, but one thing at a time.

SECURITY NOTICE:

This is a kernal mode VXD used to preprocess scan codes from the keyboard. It can read every single key press and release, every mouse movement, and take snapshots of the video screen. The vxd driver can be abused by a 3rd party program to steal your password. Hell, with a kernal mode keyboard logger, it's a trivial task. I'm letting you know that this program includes a kernal mode keyboard logger and stuffer. You decide if you want to run it on your system. BTW, dynamic vxd's dont work on NT and probably not on Win2k for OBVIOUS reasons.

Nuf' said.

Secret macro dupes? No dupes. Just macroing capability. I *wish* I had some working dupes...

KBHACKER.VXD -- dynamic vxd to stuff keystrokes into the KBC for playback. Foward all questions to your local bit bucket because I won't be answering any "how do I compile this" questions.

Code to capture scancodes, handle mouse macroing, VFLATD video screen capturing, and the *controlling Win32 app* is not shown.

There's no dependence on any API's, no interrupt chaining, nothing detectable by the program or OS for that matter, except a few OUT's to the KBC. Works with every program every made because the playback simulates real keyboard keystrokes, right down the generating an interrupt.

Ben at UO Macros (drtwisters has a link to him) is working on the same type of program that I am. I'm curious how he did his kb scancode stuffer..

BitBandit


COMMENT # *************************************************************

KBHACKER.ASM -- Dynamically Loaded VXD for Win9x
Copyright © 1996-2000 BitBandit
All Rights Reserved Worldwide

********************************************************************* #

.386p

;*********************************************************************
;
; Include files for VXD creation
;
; VMM.INC is required for all VXD's
; pull in other inc files as needed
;

INCLUDE vmm.inc ; vxd macros so it's a required file
INCLUDE vwin32.inc ; DIOCParam structure definitions


;*********************************************************************
;
; Local constants, structures, and defines
;


;*********************************************************************
;
; Virtual Device Declaration
;
; DriverName, MajVer, MinVer, CtrlProc, DevNum (MS assigned but we use undefined_device_id)
;

; 4.23 fixed devioctl close problem
; 4.15 took out 'extra' stuff. vxd is 20% smaller
; 4.11 fixed scancode playback. vkb api sucks
; 4.00 took the ver number from the template
Declare_Virtual_Device KBHACKER, 4,23, KBHACKER_Control, Undefined_Device_ID


;*********************************************************************
;
; LOCKED Code Segment
;
; Code placed here will not be swapped out
;

VxD_LOCKED_CODE_SEG


;*********************************************************************
;
; CONTROL Procedure
;
; We only handle a few dispatch messages, dynamic init and exit is
; required for every dynamic vxd, W32_DEVICEIOCONTROL to receive
; IOCTL commands from Win32 apps, and any hooks we need to install.
;
; All the IOCTL functions are dispatched again inside of KBHACKER_DeviceIOControl
;

BeginProc KBHACKER_Control
Control_Dispatch SYS_DYNAMIC_DEVICE_INIT, KBHACKER_Dynamic_Init
Control_Dispatch SYS_DYNAMIC_DEVICE_EXIT, KBHACKER_Dynamic_Exit
Control_Dispatch W32_DEVICEIOCONTROL, KBHACKER_DeviceIOControl
clc
ret
EndProc KBHACKER_Control


;*********************************************************************
;
; W32_DEVICEIOCONTROL message handler
;
;
; On entry ESI points to DIOCParams
; Pull dwIOControlCode value out of structure and process as necessary
;
; Return code goes in EAX
; use code 50 (ERROR_NOT_SUPPORTED) for unknown dwIOControlCode values
;

BeginProc KBHACKER_DeviceIOControl,

;
; load DIOCParams.dwIoControl into EAX
;

mov eax, [esi].dwIOControlCode

;
; handle DIOC_OPEN (0x00000000) and DIOC_CLOSEHANDLE (0xFFFFFFFF) requests
;

cmp eax, DIOC_OPEN
je short DeviceIOControlOpen

cmp eax, DIOC_CLOSEHANDLE
jne @F

DeviceIOControlDone:
clc
ret

;
; IOCTL init -- per app init
;

DeviceIOControlOpen:
xor eax, eax
jmp short DeviceIOControlDone

;
; IOCTL cleanup -- per app cleanup
;

DeviceIOControlCloseHandle:
mov eax, VXD_SUCCESS
jmp short DeviceIOControlDone


;
; handle all other IOCTL requests (0x00000001 thru 0xFFFFFFFE)
;

@@:
cmp eax,1
jne @F
call StuffKeyboard

;@@:
;cmp eax,2
;jne @F
;call dwIoControl2

;
; unsupported IOCTL request, return value = ERROR_NOT_SUPPORTED
;

@@:
mov eax, 50
jmp DeviceIOControlDone

EndProc KBHACKER_DeviceIOControl


;
; SYS_DYNAMIC_DEVICE_EXIT message handler
;
; Required for every dynamic driver. Driver cleanup code can go here.
;
; If there's problems with cleanup, set EAX=0 and STC, then exit.
;

BeginProc KBHACKER_Dynamic_Exit
mov eax, 1
clc
ret
EndProc KBHACKER_Dynamic_Exit


;
; STUFFKEYBOARD -- IOCTL.dwioctl=0x00000001
;
; Pushes a byte out ot the keyboard controller forcing it to create a HW interrupt
; when the KB controller echo's the byte.
;
; ESI = DIOCParams, ESI must be preserved across call
;
; cbInBuffer scancode to stuff (one byte)
;

KBLOOP equ 300 ; # of times to try each KB IO request

BeginProc StuffKeyboard
pusha ; save all registers

mov ah,[esi].cbInBuffer ; byte to stuff into keyboard controller

xor bx,bx ; BH is outer loop, repeat up to 10 times to make key stick
; BL is success bit, not zero on success
tryagain:
mov bl,0 ; reset success bit

cli ; no interrupt allowed while we're playing
; DO NOT KEEP DISABLED FOR TOO LONG

mov ecx,KBLOOP
@@:
in al,064h ; KBC status register
test al,1 ; test if output buffer full
loopnz @B ; loop until buffer isn't full or tried KBLOOP times
test al,1 ; is output buffer full?
jnz kbfail ; yes, fail

mov ecx,KBLOOP
@@:
in al,064h ; KBC status register (redundant, I know)
test al,2 ; test if input buffer full
loopnz @B ; loop until input buffer sint' full or tried KBLOOP times
test al,2 ; is input buffer STILL full?
jnz kbfail ; yes, fail

mov al,0d2h ; magic!
out 064h,al ; send to KBC command port

mov ecx,KBLOOP
@@:
in al,064h ; KBC status register
test al,2 ; test if input buffer full (i.e. wait for command to be processed)
loopnz @B ; loop until buffer is processed or tried KBLOOP times
test al,2 ; is input buffer full?
jnz kbfail ; yes, fail

mov al,ah ; get scancode to push
out 060h,al ; stuff to keyboard controller
inc bl ; set success bit
jmp kbworked ; all done

kbfail:
in al,060h ; read back from keyboard buffer (otherwise hw INT w/ bad value)

kbworked:
sti ; allow interrupts (allows KBC and VKB to process outstanding KB interrupts)
or bl,bl ; test for success
jnz allgood ; exit out if everything worked
inc bh ; inc outer loop
cmp bh,010h ; retry up to 16 times
jb tryagain ; if we don't succeed, try again

allgood:
xor eax,eax ; clear return value
mov al,bl ; 1 if success, 0 if failure
mov [esi+32],eax ; store return value
popa ; restore registers
ret ; get out
EndProc StuffKeyboard



VxD_LOCKED_CODE_ENDS


;
; INIT CODE SEGMENT
;
; Code here will be discarded after the INIT message is processed
;

VxD_ICODE_SEG

;
; SYS_DYNAMIC_DEVICE_INIT message handler
;
; Required for every dynamic driver. Driver initialization code can go here.
; Combine with SYS_DYNAMIC_DEVICE_EXIT if there's no init and cleanup.
; If you combine with Sys_Dynamic_Device_Exit, move it out of VxD_ICODE_SEG...
;
; If there's problems with initialization, set EAX=0 and STC, then exit.
;

BeginProc KBHACKER_Dynamic_Init
mov eax, 1
clc
ret
EndProc KBHACKER_Dynamic_Init

VxD_ICODE_ENDS

END

Decaying Skeleton 		posted 05-09-2000 06:49 AM         
Download 98DDK, not platform2000. 98ddk includes MASM and all necessary compiler tools to create VXD files from *.ASM source.

(install the VC++ software BEFORE the 98DDK) Go into one of the sample file, any one, build it using nmake from the dos command line. Make sure it can compile otherwise you haven't installed the 98DDK properly.

Search the VC/98DDK combined documentation for 'dynamic VXD' and 'DeviceIOControl'. You'll see a reference to a sample driver (asm) and project (vc++ prj). The sample comes with VC++ so people who only use the 98DDK won't have access to it. Rename their VXD to your own VXD. You make need to make subdirectories (see makefile. for target directory names) to hold your compiled vxd file. nmake from dos command line. compiling the asm into a vxd is a dos box command line process.

Next, load up the vc++ prj file. modify the IOCTL call to use the subfunction identified in your vxd (0x1 dword). Setup a timer to call the function and push scancode 0x27 and 0xA7 (I think that's the scancode for letter S or A, can't remember). Remember these are SCANCODES, not ASCII. keydown scancode has high bit clear, keyup scancode has high bit set. extended keys have 0xE0 prefix in front of keydown and keyup scancodes. Reference a good DOS systems programming book at your library if you've no idea what make/break kb codes are or how the 80x42 KBC works (like Turbo Pascal Systems Programming published by Abacus books).

nuf' said.
