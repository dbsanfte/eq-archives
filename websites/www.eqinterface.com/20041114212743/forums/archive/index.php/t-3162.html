Version 3.1 of UI Timer (http://www.horizonsoutpost.com/eq/timer.asp) is released! Additional Timers - You can now choose up to 12 timers (instead of 8).
Additional Row/Column Configurations - There are now many different ways to arrange your timers (for example: 2x2, 3x4, 5x2, etc.)
New Windows - Many new windows have been added since v3.0, including the bug window, feedback window, gems window, notes window, and friends window. There are now a total of 12 different windows to choose from.
Other Options - You can now specify how the timer bars should be animated (fill up or empty out), and the text color for your timers. As with v3.0, you can specify a custom label, duration, and color for each timer.http://www.horizonsoutpost.com/eq/images/timers_faded.jpg http://www.horizonsoutpost.com/eq/images/timers_compact_faded.jpg

That being said, your problem is most likely due to a mod installation issue. Here are some troubleshooting tips, for common problems that people have:

1) Make sure you've waited for the full XML file to download, by scrolling down to the bottom and making sure that the last line is </XML>. If you grab the source before it is fully downloaded, you can end up with a partial file.

2) Make sure you've created a directory directly under EverQuest\uifiles and that the directory name is one word only (no spaces).

3) Make sure you have saved the XML file using the correct file name. The name you should use is specified near the top of the XML file. This is the most common error that peple make, and causes the timer to not show up at all. The correct file name always begins with "EQUI_" and corresponds to the window that you generated the XML for.

4) Make sure your graphics file is also named appropriately. Again, the correct file name for the graphics file is also specified near the top of the XML file.

Once you log in, and type /loadskin FOLDER 1, what happens? Look in your UIErrors.txt file (in the EverQuest directory) and see if there is a line with "Error" in it (lines with "Warning" are normal and expected). If there is an error, post it here and I'll take a look.

Originally posted by Elwyn
Would it be possible to add a reset button? I don't really like messing with sliders and trying to match them up to the exact location of the bar. I'd rather just click and have the bar reset to zero. When it's full, I'd know the time's up.
Due to the limitations of UI mods, what you are asking is unfortunately not possible. UI mods can only alter the appearance of the UI, not add new functionality. Therefore, the timer is not a program, it is just a collection of animated images.

Once the EverQuest UI loads up, all animations begin cycling through their frames - there is no way to pause an animation or force it to start from a particular frame at a particular time. Reloading the UI will reset an animation, but it also resets all the animations at the same time (no way to reset one individually), and it can take a while on some systems. This was the main complaint many people had with the digital timer, since reloading the UI was the only way to reset it.

The bar-style timer was introduced to address this limitation. Because individual animations cannot be reset, and cannot reset without a lengthy reload of the UI, the timer needed to be designed in such a way that allows the player to keep track of time while the animations continue cycling indefinitely. That's why this timer uses a slider, to allow the user to mark the bar instead.

Player-made UI mods are very limited in what they are able to do, due to the limitations placed on them (they can only modify appearance, not add any functionality). Therefore there is no way to "hook up" a timer to an in-game event. The timer keeps running, and when you cast mez, you can do one of the following things:

- If using the bar-style timer, use the slider to mark the current position of the bar. It will take the bar exactly one full duration (length of mez) of the timer for the bar to reach the slider again.

- If using the dial-style timer, use the tick marker to mark either the current tick or the ending tick. The dial face represents one minute, with each tick representing a 6-second in-game tick.

- As an alternative, since mez is very short in duration, you can simply "eyeball" the timer and keep a mental note of it. This will work for bar-style, dial-style, and digital-style timers.

As far as multiple mezzes go: I usually cast each mez in succession, so I use one timer to time the first mez in the series. Then, when the timer lets me know that I need to refresh the first mez, I then proceed to refresh all of them in sequence. Alternatively, you can also create additional timers for these if you so choose.

How would one go about modifying it to be a bit shorter?

The timer bar is currently 120 pixels wide. So there are 120 frames in the animation - each frame's duration is the total duration divided by 120 (down to millisecond granularity). Extra care is taken to ensure all frames add up to exactly the full timer duration even when rounding needs to occur. In addition, the maximum duration per frame is limited to 10 seconds, so any frame that needs to have a longer duration must be broken down into multiple frames (for example, to create a 25-second frame, you must make two 10-second frames and one 5-second frame using the same image).

To make the timer smaller in size, you would have to have fewer frames (as the timer already uses one frame per pixel), and adjust the duration accordingly for each frame. In addition, there are other changes needed to adjust the size/position of the other elements (slider, etc.). It is probably too tedious of a task to perform by hand, especially modifying the animation frames in the XML file. In a future release of the web-based XML generator, I will consider adding options to change the size of the timer.

Ever since I released the original tick timer, the same requests for "impossible" enhancements have come up repeatedly. To save myself some time, I've decided to write up a detailed post on what isn't possible in the timer mod and why, so I can answer all of these questions with a single post.

The information in this post is not new; these questions have been answered and discussed previously in this forum. This post is just a consolidation of those topics, which can be found in the following places:
- Several posts in this timer thread
- Several posts in my other three timer threads (linked to the first post of this thread under "Old Forum Threads")
- The timer FAQ on my website
- Various posts in other threads here at EQGUI (mostly in the request forum and developer discussions area)

Impossible enhancements

Here is a list of requests I frequently receive. If you have a question along the lines of the ones listed here, please read this entire post carefully, as it will explain why your request is not possible.

Can you add a reset button that restarts the timer?
Can you add pause/resume buttons for the timer?
Can you add a button that moves the slider automatically to the position of the bar?
Can you start the timer automatically when I cast my spell or use my ability?
Can the timer resume from where it left off between sessions?
Can the timer show system time, or Norrath time?
Can you add new commands (like /timer start) that let me control the timer with macro hotkeys?
Can you add functions/variables I can access with my own UI mod?
Can the timer be set up to automatically execute in-game commands at specified intervals?

Unfortunately, the answer to all of these questions is no. These enhancements are impossible due to the limitations of UI mods.

Player-made UI mods

First, I'll start with some general information on UI mods for those who are unfamiliar with the capabilities and limitations of UI mods.

Player-made UI mods can only alter the appearance of the UI, they cannot add any new functionality. So, you can change the size, color, position, etc. of various elements in the UI, and you can add things like text labels and images (including animations - more on this later) to be displayed, as well as standard UI controls. However, you cannot add any actual functionality, such as new commands, or new buttons that execute commands. Anything that has functionality in the UI is hard-coded in the EverQuest client (i.e., only Sony can add new functionality).

Another important related point is that UI mods are not "programs". There are no function calls, no variables, no way to execute any commands or detect when particular events occur. UI mods consist of only two simple parts: XML data files, which contain information loaded by the EverQuest UI engine, and graphics files, which contain images used by the mod. The XML data files must contain only those nodes allowed by Sony's pre-defined schema, and they specify the various attributes of the UI's appearance only (see previous paragraph).

Animations

An animation is basically an image composed of multiple frames, which are displayed in sequence. Each frame consists of a fixed duration and an image that is displayed for the extent of that duration. An animation always starts with the first frame, and continues through its list of frames. After the last frame has been displayed, a repeating-type animation will start over from the first frame.

As soon as the EverQuest UI loads up (when you log in or whenever you type /loadskin), all animations in the UI begin displaying each of their frames in sequence. There is no way to pause an animation or force it to start from a particular frame at a particular time. Reloading the UI via /loadskin will reset all animations, causing them to start over from the first frame. This, however, freezes the display temporarily and takes too long on some systems, and does not allow animations to reset independently of each other.

UI Timer

Based on the above information, one can see that my timer mod is not a program at all - it is nothing more than a collection of animations. Depending on which type of timer you choose, each animation is either a bar, dial, or numerical digit. These animations cannot be controlled and have no connection to in-game events.

Due to the limitations and lack of ability to control animations, the slider and tick marker were introduced for the bar and dial timers, respectively (these markers must be manually set, as they have no awareness of or connection to the animations). These give the user the ability to manually "mark" the animation, allowing you to begin timing at any point during the animation's cycle. With these, the timers can be used without mentally keeping track of when timing started or resorting to reloading the UI.

I hope this post has been helpful for those who have been wondering about possible enhancements or why the timer is designed this way. If you have any additional questions please feel free to let me know.

To fix this, just generate the file again, but wait and make sure the full file is finished downloading before you save it, by scrolling down to the bottom and making sure the last line is "</XML>".
