Ok, answer to question 1:
If you changed the size of the graphic (made it 200 pixels wide, is this what you mean by change the size?) you'd have to make the "Ui2DAnimation" numbers and the "Gauge item" values bigger also.
The reason the numbers are so big is you want to show only portions of the graphics, so you have a 10000 pixel wide gauge but only show a 100 pixel wide fragment of it. The reason the gauge's values are similar is because they are similar gauges, they're just the different colors.

Q2:
The tabs are used to constrain the left and right boundaries of the gauge so it's only 100 pixels wide. I've, since the posting of that code, developed a cleaner method for multicolored gauges that doesn't need tabs. They were not just left overs, they were necessary. I'll post the new code if you'd like it. It cuts back the required code for a multicolored gauge significantly, or rather, the complexity of the code, probably runs a tiny bit better, performance-wise, also.

Clarify something for me for question 1: when you say change the size of the graphics, do you mean change the width and height of the graphics file or change the size of the graphics for the gauge?

If the width and height of the graphics file change, this doesn't matter so long as the gauge graphics are 100 pixels wide, 8 pixels tall and a the XY location 110, 20. When you have a 256x256 pixel image, and you ask for something at XY coordinate 256, 0, it would grab the pixel at 0,0. EQ wraps the graphics files when you query for data past the edges. That's why I can grab a 10000 pixel wide slice of graphic from a 256 pixel wide file.
