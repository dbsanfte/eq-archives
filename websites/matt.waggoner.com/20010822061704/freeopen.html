What's the deal with this Free Open Source Software Linux thing?

It seems that a large number of people have taken to the idea of software that not only costs no money (which makes it "free" as in "free beer" and "free admission", relating to the Latin root gratis), but whose source code is available for anyone to take, modify, and even SELL if they feel like it (which makes it "free" as in "free speech" and "freedom", relating to the Latin root liber), so long as that right is maintained.

No, not like freeware. Freeware is simply software that costs no money; but it is rarely (if ever) also free as in free speech. Free Software is software, such as the Apache web server, that anyone can take, modify, distribute, and sell for money (or even give away for no money if they feel like it).

The difference between Ford and, for example, Apache, is that the Apache software costs virtually nothing to duplicate -- once you have a copy of Apache, you can make infinite perfect duplicates just by, well, copying the files. And because the people who created Apache did so using a Free Software software license (specifically, the GNU General Public License), anyone who wants to can take the Apache software, modify the source code, and distribute that modified version -- they may even sell their modified version of Apache for money (any price they feel like charging!), as long as they follow the terms of the GNU General Public License.

The method by which pieces of software like Apache and Linux are developed is something called Open Source. This is not the same thing as Free Software; they are related but separate. Open Source is the idea that you make the source code of your software available to the public--

Wait, what? If you give everyone the source code they can just steal the software without paying for it!

Like I said before, they aren't charging for the software, so it doesn't matter if people "steal" (or more accurately, copy) the source code. They WANT the source code to be available to anyone who wants to look at it!

Why?

Because if you only have the 10 programmers at your company look at a piece of software, then you are limited by the abilities of those 10 people. If you make the source code available to anyone to use, then you suddenly have hundreds or thousands of people looking at the source code, for one reason or another. And because they want the software to improve as much as the main developers do, they'll be sure to point out a bug they found, or even submit patch code themselves so you don't have to take the time.

Well that sounds like a good thing. I guess. But wait, how can thousands of people possibly coordinate to write a piece of software? Won't there be conflicts?

There would be, if thousands of people were allowed to directly modify the source code to a project; however no Open Source (OS) or Free Software (FS) project actually works like this. Almost all projects have a small core of developers who are the main programmers for the project -- they do most of the primary code, as well as decide which of the submitted patches get used.

So how is it "open" or "free" if only a small group of people actually get to work on the code?

This is a very common question about OS/FS, and is often used by opponents of OS/FS (or people who just don't get it yet) to try to denigrate the abilities of OS/FS. So here's an explanation.

Let's take an example; the one I like to use is the OS/FS project Freeamp, which is an OS/FS music player. (It's kind of like Winamp, except Winamp is owned by AOL and is neither Free nor Open. It's free in the beer sense, but not in the speech sense.) Freeamp only has a handful of developers who are allowed to commit changes to the source tree. So then it's an obvious question, well how is that free or open? The answer is thus: Freeamp's source code is ALL AVAILABLE TO THE PUBLIC. ANYONE is allowed to download an entire copy of the source code and do with it what they want. Which means that if I don't like the direction Freeamp is going, I can go download the source tree, change all the instances of "Freeamp" to, say, "MattAmp", and start my own project. Because of the license Freeamp is developed under (The GNU General Public License, or GPL), I am allowed to do this. I am allowed to modify the Freeamp source, distribute the modified source, and so on.

Let me see if I've got this straight. Even though only a small number of people can actually make changes to a particular project, the openness/freeness comes from the fact that anyone is allowed to take all the work that's been done so far, and start their own project?

What's the difference between Free Software and Open Source?

This question requires a little bit of background. In 1984, Richard M. Stallman (commonly referred to as RMS) founded the Free Software Foundation. The goal of the FSF was to create an entirely Free operating system called GNU. You can find out more about this at the FSF page; I won't try to repeat it all.

The upshot was, Stallman created something called the GNU General Public License (GPL), and all of the FSF's software was released under the GPL. The FSF went about creating all manner of common Unix-like utilities; ls and ps and fsck and just about every Unix utility program you can think of.

The GPL basically does the following. If you release a piece of software under the GPL, it means that if you release compiled binaries of the software (a regular compiled .exe program, for you Windows people), you must ALSO make available the source code to everyone to whom you made available the binaries. It also says that any code derived from GPL code, also falls under the GPL. (If you write a piece of code completely from scratch, and use it with a GPL'd program (or use a GPL'd program to compile it), it does NOT "convert" to the GPL.) There's some more details but that's the gist of it; again, go to the FSF's web page for the full scoop.

As time went on, people who were not interested in making software for the sake of money (students, professors, miscellaneous other folks) began developing more and more software under the GPL and similar Free Software licenses. In 1991 a project began, started by a Finnish college student named Linus Torvalds. The project: Linux.

Are you getting to the difference between Free Software and Open Source or is this a history lesson?

I'm getting there, but if you want to really understand the difference, you've got to learn all this.

As the 1990s wore on, Linux became closer and closer to a real operating system. Eventually, by 1993 or so, you could actually use it for some decent tasks. By the new millennium, it was nearly as full-featured an operating system as any other. The entire Linux kernel and almost every program that comes with a Linux distribution, was released under the GPL.

Open Source, as a movement, was something (theoretically) invented in 1998, not by anyone in particular, although Eric S. Raymond (ESR) is often credited with being one of the founders of the movement.

Free Software is the idea that users of software should have the power to modify software to suit their needs; to make the software work for THEM instead of vice versa, or instead of the software being controlled by a company whose best interests are making money, NOT whether or not you have the software you want.

Open Source is really the method behind many Free Software projects. The idea is that the source code is open and available to the public, DURING THE DEVELOPMENT PROCESS. This means that anyone can contribute, bugs can be found quicker, and development is spread over hundreds or thousands of people instead of just a handful.

FS and OS have, in my opinion, more or less merged in the last couple of years. RMS is against Open Source because OS allows things like the BSD license, which Stallman does not like. (The BSD license is different from the GPL in that the BSD license does NOT require you to distribute the source code to the software if you modify it. You can take a BSD-license program, take a bunch of the source code, and use it in your closed-source program, then sell the program, and you are NOT required by the license to release the source code to your program.) However, FS and OS have, like many words in the English language before them, both become corrupted. OS never had One True Meaning; it was more of a collective phrase to refer to the FS/OS phenomenon in the late '90s. FS on the other hand was something strongly defined by RMS.

The vast majority of OS projects out there are under the GPL anyway, so the point is becoming mooter by the minute. Very few people have a desire to use anything but the GPL; if you're working on an open, free project, the GPL is usually the best choice.

There's also a major schism between those who favor the BSD license and those who favor the GPL. The BSD side says that the GPL restricts the rights of software developers because it requires them to distribute the source code. The GPL side says, that's exactly the point! Users are more important than developers; users need more of the power when it comes to who can do what with software.

Whew... okay... I think I'm finally getting this. What other advantages does OS/FS have over, say... well what's the opposite of OS/FS?

The opposite of Open Source would be, not surprisingly, Closed Source. There's no good opposite of "Free" unfortunately, so the opposite of Free Software is usually called Non-Free Software. But for our purposes, we can just call it Closed Source (CS).

CS software is stuff like Windows, Microsoft Office, Photoshop, Mac OS, Oracle, almost all games, and the majority of programs that have been historically available. All of this software has been developed by companies that were doing it for money, not because they wanted that software so they could use it, or because of any attempts to make the world a better place. CS software's source code is NOT available to the public; the only way to ever see it is to either be a programmer for the company, or to buy a license to see the source code (usually for thousands of dollars, if not millions in some cases).

So what's wrong with that? Those companies do pretty well.

The problem with CS software is that because the source code is not available, then if you discover a bug in the program, there is usually nothing you can do about it. You are forced to wait until the vendor releases a bug fix for that bug. Commercial vendors, on average, do not have a particularly good track record when it comes to fixing bugs. Microsoft deliberately leaves a certain amount of bugs in their software as a business practice; every so often they release a Service Pack for their various large pieces of software (Windows NT, BackOffice, etc.), that will fix a number of bugs but usually introduce many more.

In an open source environment, you HAVE THE SOURCE CODE. If you find a bug, you can fix it yourself; or if you aren't a programmer, odds are, someone else will already have fixed it, and submitted a patch back to the developers, who will then include that patch into the source tree. OS projects tend to fix bugs far faster than CS projects, because OS projects don't have to wait for the next version of the software to come out. Because of this, OS projects tend to be more stable than CS projects.
