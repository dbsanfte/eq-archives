From this I suggest the procedure following (even if I don't like it that much) :
* the first and the last point tend to say that increasing the average swing number per round doesn't increase the proc rate.
* the ppm for each proc is calculated from DEX,WA, proc rate mod and wearing hand.
* the rpm (round per minute) is calculated from weapon delay and haste (remember that there are separated for each hand).
* the chance to proc per round is calculated for each proc from the numbers above.
* theese proc rates are fixed on our character file (only updated when changing gears, buff, ...).
* during the round, the procs check is made multiple time (as much as swing check ?). but after a successfull proc, the proc rate of this particular proc is set to zero untill the next round to prevent the same proc to fire more than once and thus help other to fire once.
* during the procs checking, the check is made from the first to the last untill one fires or we run out of proc. thus we have a proc order in ppm (ex: BoC+aug gives 1.99ppm + 1.95ppm after 10H parse and BBB+aug gives 2.66ppm + 1.89ppm) and we can't fire more than one proc per attempt.


Well, this is not perfect, but it could explain theese strange logs...

Note : a fully slowed warrior wearing a weighted axe should have less than two round per minute. so he can test the proc system more accurately if he make parses between AAs (WA, Flurry, Raging Flurry, Punishing Blade).
