Human nature is a funny thing. People often attribute streakiness in randomness to be a lack of randomness, when in reality, you can not have a truely random event without streakiness.

I had a probabilty class in which the teacher had 1/2 the class roll a 20 sided die 100 times, and the other half made up 100 fake rolls...

After these were all written down the teacher looked over each of them (not knowing which were fake, and which were not) and labeled them as such. He correctly labeled all but one of them...how?? He went thru them, and the ones that had the same number appear 2 times in a row, or more, were the real die rolls, and the ones that never repeated a number were made up.

Humans don't like seeing patterns in randomness, but true randomness will indeed have patterns and streakiness.

Daabu
Gerik's triple-8's were actually different numbers every time.

Random Number Generators, at least the one I'm most familiar with (simple rand() in nearly all programming languages) actually generates a random decimal from 0 to 1 , then multiplies that number by randMax-randMin, adds randMin and truncates.

Example, I want to roll a 20-sided die.
randMax = 20, randMin = 1.

The RNG gives me a 0.44658974. Multiply by 19 gets me 8.48520506. Add 1, and its 9.48520506. Truncate, and you get 9.

Roll from 1 to 10 and get an 8 == 0.77778 to 0.88888

Roll from 1 to 11 and get an 8 == 0.7 to 0.79999

Roll from 1 to 8 and get an 8 == 0.875 to 1

The roll could have been 0.824, 0.706, 0.935...all would give 8, but none are close to being the "same"

Bonequest
With most programmed languages, if you plot the nth random number vs. the (n+1)th random number and keep doing it enough, you will start seeing a series of diagonal lines that are parallel and equally spaced horizontally and vertically. This is because "many" (not all) computer generated psuedo random numbers are in fact just one element in a huge string of numbers generated with a mod function. This string repeats itself eventually and if you see a situation where, for example, 45345 is always followed by 76128 on consecutive numbers picked, this means they fed the last number into the seed for the next number. This means that if someone could monitor the random number stream for all random numbers returned, they could build up an [n,n+1] map of the random number generator and predict the next number given a very short sequence of observed returns effectively deducing the mod function used to generate the sequence. This can be done with a program if the programmer has access to the entire random number sequence and can monitor it long enough but I don't know if this is the case in EQ. I would hope not.

Of course, this can be easily offset by programmers by mixing things into the seed like the system clock value, the number of people in a zone, etc. which are more likely to be truly random so the random number sting would not be as easily (without using a supercomputer) mapable in that case.
