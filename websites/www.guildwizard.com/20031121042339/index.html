Permissions/Security update - bugs fixed Wednesday, November 12, 2003 - 9:43:13 AM

Whoa!  Large patch was just deployed.  First let me get the bug fixes out of the way:

Bug fixes:

  • Fixed domain look-up tool.  You can now look up domains to register on the domains page again.
  • Fixed stored-procedure that wasn`t listing items purchased in raid detail
  • Fixed bug that allowed deleted members to still login to their guildwizard account.  If you delete a player, they will no longer be able to use their login/password.
  • Fixed problem with Loot List that wasn`t sorting the items with the most recent item at the top.
  • Fixed up Admin links that didn`t have style attributes added to them to change the mouse pointer to a `pointing finger` (which made it hard to realize it was a link).

New Features:

  • Each page now has a permission setting which allows you to allow Everyone, Logged In Members, or Admins to view the page.  So if you only want Guildmates to see your raid-list, you can set it so only after they login will they be able to go to the page.  If a user tries to go to a page with insufficient rights, they are redirected to the login page.  After they sucessfully login, they are brought back to the page originally requested.
  • In addition to Page Level Permissions, most pages have Item Level Permissions as well.  What this means, is you are able to set who can see particular items on various pages.  For example, you can set the whole Calendar Page to allow only Guildmates or you can allow Everyone to see the page, but certain items in the calendar would only be viewable by Members or Admins.  Items in the calendar that a user doesn`t have sufficient privledges to will show a message telling them they don`t have rights to see the message, and redirects them to a login page.  
  • All grids that were able to hide columns, have the ability to have the column visable to Everyone, Members, Admins, or Nobody(hide).  So on your Roster, you might only have the Name,Class, Race and AA points viewable by Everyone, but the DKP earned only visable to Guildmates, or Admins.  It`s the same configuration panels as before, just with the new options added to them.  Again, this functionality is extended to all pages that were able to configure grids (Roster, Purchases, Raids, Encounters etc).
  • Custom pages now have a field to set permissions to them.  When you create a custom page, you will be able to set who can see the page.
  • In the Rank column of the Roster, you can now set the rank to "A" for alternate players.  This is a quick-bandaid for now.. I want to spend some time on how alts are being handled in the Roster, now that the Guild Dump feature includes everyone in the guild.. there needs to be a better distinction.

Upcoming Features:

  • The Roster will have a mouse-over effect when you roll the mouse over a person`s name.  A semi-transperant box will appear with quick info about the player.  It can have tradeskill info, Keys/Flags.. whatever they/you want.  It will be editable by the member and the Administrator.
  • Administrators will be able to edit member`s profiles without having to login as their accounts. 

I just want to note that every single change or upcoming feature was a result of people sending me suggestions or posting suggestions on the forums.  Thanks for the input guys/gals, keep it coming!

Raid Dump file added, cache for raids changed Monday, October 27, 2003 - 8:21:00 PM

As promised, I added support for the new (yet still broken) raid dump file in the Raid console.  I also added a help page for the Raid Manager page so those who have never added a raid can learn how to do it.  The new file is dumped to your eq directory when you click the Dump button in the Raid Console.  It creates a new raiddump-YYYYMMDD-TTTTTT.txt file  (Year/Month/Day-Time).  Don`t change the file name, as I use the file name to parse the date/time of the raid (since it doesnt provide that info in the file itself... /bird SOE)

I left in support for the /who list cut/paste.. you just have to click the checkbox to enable it.. when the Dump thingy works in EQ though, use that.. its better and more reliable.

I also changed the cache to update immediatly after you enter a raid.. It was causing too much confusion with it adding later, people were thinking the raid wasn`t added at all.. which is understandable.  So now when you add a raid, it will show up right away in your Raid List page.

Raid Dump Feature Coming Sunday, October 26, 2003 - 9:44:20 AM

The raid dump feature that was recently added to the Raid Management console, which right now is broken, will be added to the Raid insertion page.  I am going to continue support for the present method of insertion (the fugly /who list), but I encourage everyone to use the raid dump feature instead.

Up until the dump feature was added in EQ, the who list (or raid channel dumps etc) was the only way to parse who was on the raid.  This new feature will create a text file of everyone on the raid, their group #, and the group leaders.  Only the name will be parsed.

I have the component finished, I just have to write the database parameters to handle the new component.  So look for another message later today, tommorow at the latest.
