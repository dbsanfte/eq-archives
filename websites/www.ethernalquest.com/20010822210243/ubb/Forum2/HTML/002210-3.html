Okay I'll *try* to keep this post oriented around design instead of features

Going back to the issue of inheritance - the more I think about it, the more I like it. It would solve issues such as creating two NPC's that are virtually identical, except maybe with a different name and 100 more hit points. This way, you could avoid duplicating a lot of data. Example: two guards with identical data, differing only by name and roaming paths.

Now, I realize that this is a rather more advanced topic and doesn't need to be done right now - it can always wait till later. But with proper design, it can be easily added at a later date, as long as the overall design supports it (or I could do it myself if the emu goes open source! )

Borrowing your earlier Froglok example:

Define NPC "Base Froglok"
{
Level = 5
HPs = 150
Appearance = whatever
Equip = blah blah
blah blah blah blah
}

(Borrowing a keyword from Java...)

Define NPC "Badass Froglok" extends "Base Froglok"
{
HPs = 20000
}

When the script parser runs into the "Badass Froglok", it will see the "extends" keyword, so it looks for the "Base Froglok" script, initializes the NPC with data from the base first, then override them with data from the Badass. You've just avoided having to needlessly copy all the base data and increasing the size of the script.

A couple of issues...

1. I'm not too keen on the idea of using string literals to define npcs/items and using them as arguments in scripts *and* using them as names that clients will see over their heads. What if you made a typo/spelling error that you didn't notice until later on, by which time, the object name would have been used in a bunch of different scripts? You'd then have to hunt through all the scripts that uses it to correct the typo/error. Yet, having an ID system is probably going to be too confusing and obscure for most people. The cleanest way I can think of to resolve this issue is to simply use "GayFroglok" as the classname, and to have a:

Name = "Gay Ghoul Froglok"

field inside the class. This way, you can change just the name if you want/need to without touching the class name and therefore messing up a bunch of scripts.

I suppose you could even make the Name field optional, so that if it doesn't exist, then just use the class name as the npc name... if a typo/error is found, just add a Name field! (oops, I guess this *is* a feature, not design. Oh well)

2. For the "Level = 29" field.. does that number ever change? I'd like a bit of randomness... so maybe represent the number instead as XdY+Z.

Examples:

0d0+500 = a number that's always 500.

5d5+0 = a random number between 5 to 25. (for you non-D&D players.. that means 5 throws of a 5-sided die and adding up the results).

5d5+10 = a random number between 15 to 35 (get result of 5d5, then add 10).

Warning.. this post is too long. sorry.
I'm actually pretty eager to see the next release. I may have mentioned this here and there, although I purposesly never mentioned it much while I was at HackersQuest, that I had been working on an emulator before I ever joined. Simply because I had no plans of ever releasing mine it was just for fun. To see if I could do it, which I think in many areas Ashran has me beat hands down [specially like using packet sniffing to gather item info, mob info... I just didn't think of doing that.] Just as a foremention, these are just suggestions. This is what I did, but I am not going to give out any code or give out anything at all. I don't really even like talking about it, but this is a good discussion so I wanted to just mention some stuff.

The emulator was called "sEQuel" (corny, I know). I primarily focused on the "setup" of the server and not the actual content of it or how fast the network code was or even if it was playable really. I could spawn stuff while in game, except items which had to be predefined onto a character before the server generated the player and my items were quite extremely lame [since I didn't have a handy dandy item.db]. Anyways that's all playability, so...

The 'client' program was launchEQ. You run that program and a window opens up asking you for your login and password. This would determine if you were any of the admins or a common user. When it was submitted it would check it against the u/p in a MySQL DB on the network [or internet as the case may be]. Once logged in, it would pop up another window which queried the db for available 'world' servers. It would return the name of each 'world' server, the number of people online, the current zone stability [which was based on percentage of zones up at that point in time], and [because I'm stylish like 'dat] a small 50x50 image representing the server. When you highlight the server, on the right it would load that image as well as some information and a short description [all of which was pulled from the db]. [alot of times when I say pulled from the db, I really mean saved to a .tmp file in the EQ directory and then pulled from there, btw.] You click login, lanchEQ sends your server selection to the DB, then edits the proper files and your EQgame connects to the emulator login server which then connects you to the proper world server. The rest from there was bug-ridden madness.

The server side was sEQuel. It worked like this. I had 7 computers on the network at the time.
myth01 - Computer I used to play EQ with
myth02 - 'data' server. Has the MySQL db, login server, and modified EQ files.
myth03 through myth07 - zone servers.

Just real quick, a "world" was nothing more than paring up of a few computers. For example the data server would have in the db that myth03, myth04, and myth05 (that's at least 3 zones) was one world called 'Entropy Server' and that myth06, and myth07 (that's at least 2 zones) was another world called 'Chaos Server'. So if you were to use launchEQ it would show Entropy Server and Chaos Server. So in other words the data server had something similar to this in the db...
1 - myth03 - serv8291 - Entropy Server - South Qeynos - 0 players - active - ...
2 - myth03 - serv8291 - Entropy Server - North Qeynos - 0 players - active - ...
3 - myth04 - serv8291 - Entropy Server - Plane of Poo - 0 players - active - ...
4 - myth05 - serv8291 - Entropy Server - Plane of Pee - 0 players - active - ...
5 - myth06 - serv9999 - Chaos Server - South Qeynos - 0 players - active - ...
6 - myth07 - serv9999 - Chaos Server - North Qeynos - 0 payers - active - ...

So you can see. The first entry is just an id #. The second entry is the computer addy. The third entry is the world server id. The fourth entry is the world server name. The fifth entry is how many players, then wether it's active, and so on. This isn't how I had it setup, but it's an example. Mine was actually pretty different from this. Under another table, btw, are the world servers by themselves. This contained all the characters and their location. So when someone connects to myth02 (the data server / login server) then tries to hit "Play EverQuest" and connect to their world, it searches the db table for the world they chose to find out where their character is then sends them there. So if it finds that their character "Draigon" for the "Entropy Server" last logged off in North Qeynos, it would send that user to myth03. [at least that's how I wanted it to work in theory... I did something much different which worked, but sucked hard.. not gonna go into it.. it's embarassing]

Just one last thing to this way too long post...
myth02 (the data server / login server) contained logs. For each of the zone servers, every few minutes it would send back a report to myth02. Updating all the stats (like players online), but also sent log files which was then appended to the log file for that server. This worked out realy nice for debugging, although wouldn't be practical for actually running a server like this. Would be too slow.

I probably explained this like crap because it's mixed in with how I wanted it to work in theory and also some of what I actually was able to do. So if a part of it doesn't sound like it would work, it's likely that it wouldn't. I quit the project about 4 months after joining HQ which was what... Dec 99?

The zone object could contain some type of array or linked list of NPC pointers for each NPC in the zone. Now, to "zone" an NPC from, say, lavastorm to soldungb, is 1. We probably wont use any inheritance in the definition of NPCs, etc because it too much of a pain in the booty and striaght data is more straightforward, however, I dont think I'll want to run into your "Badass Froglok" anytime soon.

As far as the other stuff, scripting is still off in the distance to some degree. And there may be a possability of this being database driven to ease editing (meaning that there'll be an app that connects to the database to edit the stuff). But as a rule, it appears that NPCs with a given name are a unique spawn, and this is why I proposed using the name of the NPC as the unique identifier for the spawn (because that's how it appears to be now). Of course, each spawn can have a level range, etc.

To further on that, and answer somebody elses question, when a defined spawn turns into spawned spawn, the internal name becomes the spawn name with spaces convereted into underscores and a 2 digit serial number at the end.. thusly, the all-familiar "a_decaying_skeleton01" Also, keep in mind that when you say "5d5" you aren't specifying a 100% random number between 5 and 30. The number is weighted towards the middle because the forumal is 5 random numbers between 1 and 6 and the odds of a 5 or 30 resulting are extreemly rare (one in 7776 for each) while the odds of a 17 or 18 are very likely (less than 1 in 25). This is called a "weighted" random number. I prefer straight random numbers because they are more clear when you spell out odds for something... as in my prposed RANDOM statment:

RANDOM(weight, result [, weight, result])

where weight is either a number indicating weight or a percentage.

as far as the scripts being loaded, you wont have to reboot the server, but yes, you will have to stop the zone server and restart it.

directory structure is a long ways from our minds at this point. But I imagine a single script file all spawn definitions in a zone, and a separate one for NPCs.

Keep in mind that our PRIMARY goal here is to EMULATE and not recreate. So we're putting our energy into relicating live EQ servers and not into building a game engine for creative people to make up their own EQ-like games. Besides, with what we end up with, you'll be able to do all this stuff anyway. As far as the pluses you gain from stuff like inheritance, this can be implemented in an editor that does it for you, and not in the language it's self.
