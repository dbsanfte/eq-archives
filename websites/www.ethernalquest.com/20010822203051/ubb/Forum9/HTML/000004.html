Innoruuk 		posted 07-28-2001 02:17 PM               
There are a ton of reasons, I'll outline some of them here

All events in a MMORPG must happen In Order.

Changing to the Shotgun, the firing is not the same as Firing, the switching to the Shotgun.
Changing target, then attacking is DEFINITLY not the same as attacking, then changing target.

You can "overcome" the problem of 'attacking and switching target' by letting the 'attack' packet include the target.. And the posision you're in.. The position you believe the target to be in.. The weapon you have selected.. Your movement at the time.. Hmm.. Simply firing your gun simply became a 2 KB packet
Hence, it's important that the server and client agree on the exact state of things, otherwise you can't do simple delta updates.

For FPS games and the like, things like these aren't really that important, as the 'fire' packet includes your gun # (which is a fixed # from 1-10 usually) and your firing vector. Whether you hit or not is done serverside.

Further, all data must be acknowledged in some form. For example, skill updates, the message that someone is attacking you etc etc, those are VITAL, it's imperative that they get across.

A MMORPG has to cope with 5% constant PL without being totally unplayable. UT, Q3 etc are unplayable with 5% PL.


Alternately, you can design both client and server so that they can survive with lost data by using prediction and other fuzzy means to understand what the client or server really meant and what is likely to come next.
So what happens if my group attacks a mob and I start lagging.. The server assumes I want to continue casting nanoformulas... Which I don't, because I want to keep my nano, and I don't want aggro.
Prediction only works properly for FPS clients, it's damn hard to predict what a RPG player really wants to do


But let's, for the sake of argument, assume that they actually DID all this; added predictions and made both client and server happy in packetlossy situations... And along comes Joe Hacker and starts chancing packets, so you have to add encryption.
Dayamn.
Since you use UDP, and you have elected to survive packet loss, you can NOT chain the encryption from packet to packet, and it's a trivial thing to break it.

But I'd like not to think of "hacking" as a reason, as in a perfect world there would be no such thing, so let's take another form of stream manipulation: Compression.
Stream-based compression is about 40-50% more effective than packet-based ones, on the mere basis that you can keep your wordlists and dictionaries across the small packets. For large events in a MMORPG (in which there are 30+ players), compression is a Damn Good Thing.

.. So what you end up with is something like EQ, in which you use UDP because everyone says it's good and less laggy, and you add your OWN routine to acknowledge packets and request retransmission when they don't arrive.
Oh, and if you send so much data down the link that it's so lagged that it gets retransmitted before it gets there in the first place so you just add more lag, that's just a feature, right?
Basically, you end up with the disadvantages of both UDP and TCP without any of the advantages


For online MMORPG, where each data stream is unique to each client and all events must be acknowledged and recieved in order in both ends and probable prediction is hard if not impossible, TCP is the only sane choice
Using TCP for Anarchy Online is one of the things that prooves Funcom actually KNEW what they were doing when they wrote it; they broke the current trend of copying the FPS network style and chose to use one suited for a MASSIVE multiplayer game.

Innoruuk 		posted 07-29-2001 11:10 PM               
TCP has no limitation of 65536 streams, what gave you that idea?
TCP is connection-based though, so each connection takes up a file descriptor, meaning you eventually run out of those on the host OS, but most REAL operating systems can cope with several hundred thousand simultanous connections (in theory).

Yes, TCP is packet based as well, but it already has things like..
Fragmentsize dependant on packet loss
Windowsize dependant on speed and packet loss
Dynamic speed reduction upon congested routers (ECN).

Meaning it was actually designed and has all the functionality to cope with the hundreds of different connection scenarios on the internet. It does it's best to give you the information as fast as possible, even if parts of it is constantly lost.

Yes, EQ could be redesigned for using TCP, but there are several other things I'd do first if I were them

It's true that TCP normally has no 'lag indicator' and the like, but most operating systems allow you to peek at the TCP socket statistics, giving you those exact numbers (RTT, PL, Windowsize and so on).

The bottom line is that if you are More of a Packet God than the people who made TCP in the first place, then use UDP and do your own acknowledgements, windows, dynamic resends and all that, but if you're not, and it's important that the data get there, use TCP.

Decaying Skeleton 		posted 08-01-2001 07:19 AM            
TCP is basically a rich featured UDP protocol. The underlying mechanisms for both are basically the same, but TCP guarantees ordered, reliable transmission. Problem is that most client/server models force the server to ack client actions, so a hacker doesn't insert 200 attack messages into his bitstream. A UDP sliding window protocol (basically what EQ and TCP use) allows for a natural way to do this. The problem is that TCP will handle all of the ack'ing internally, so the program will not see it. Hence if you want any synchronization between what's going on between the client and server, you need to implement another acknowledgement method on top of the one that TCP is already using.

Next problem with TCP is that since its sent as packets, it has a send/receive window which is how far ahead the transmitter is allowed to send packets before it has to block and wait for the receiver to sort out which packets it received and request retransmissions of missing segments. A UDP protocol can continue to send current information, so the client will have a partial view of what's going on. A TCP protocol will block, and the user will see nothing. When the stream finally starts moving forward again, it has to process through old data that quickly, so there will be a period where the client is totally out of sync with the server as the server will have stopped sending current info due to a full send window. A UDP protocol will keep blindly firing packets away regardless of the situation over the network.

The internet is primarily built on packet switched networks, thus TCP and UDP are both packet based as was said before. TCP is easier to use, but less flexible. UDP is much closer to what is actually sent out over the wire.

I don't agree that TCP is better for MMORPGs than UDP, in general you will get better performance with UDP in all real-time games.

The point is that TCP is just a Transport protocol, the purpose behind most of the algorithms used by it is to allow as many as possible medium to large data transfers through the bottlenecks of the internet.

In other words, it's considerate of all other traffic, and if the developer (Funcom in this case) has enough bandwidth for it's customers (which it should) then any packetloss will be caused by other unkown internet users. Why should a real-time application for example use TCPs AIMD algorthm? If only 10% of your packets get through you shouldn't just bend over and say "Oh no, I'll back off and let you download your mp3 archieve before I send my crucial realtime data Mr. unknown_surfer023", you should send as much, if not MORE packets to MAKE SURE your crucial data gets through.

Yes, some might find it inconsiderade, but you pay for that bandwidth, and unless all the users are on overloaded networks they should only be a small percentage of all users.

As for the discussion on in order events, just sending your target and your weapon would do fine, any calculations must always be done on the server anyway, I can hardly see it becoming 2k. Also, the added overhead of the TCP protocol kind of makes that discussion a moot point.

If you wanted to you could implement the whole TCP protocol process in UDP just by reading a couple of RFCs (contrary to common belief they are overall quite readable), but as mentioned before TCP is generally too cautious to be effective.

As for ECN it is as far as I know not publically used except in a couple of test networks. It also requires that the users have an updated TCP stack to actually use it.

Slicer - You point at TCP's methods for dealing with congestion as a good thing - Its not - Exponential backoff for an interactive game is definately a problem.

Games, whether MMORPG or FPS rely on a certain degree of interactiveness - The actual quality of much of the data (ie, whether all of it turns up) is of lesser importance, in many cases. TCP connections are good for stream downloads where getting *all* the content (rather than 'enough') is more important than whether it arrived in less than a second or so after it was sent.

Regarding Compression... True decent streams based compression does do a better job than basic per-packet compression, but at the cost of more intensive CPU usage - Not significant on the client really, but it could begin to hammer the servers. Also, a well designed packet based system will often be very close to the actual transfered size, and also has a tendency to be *lower* on cpu usage than 'fatter' systems.

Now you do have a point in terms of encryption being notably better for streams based systems vs. packet based, but the differences are pretty small given a decent algorithm and implimentation. However, I must point out that encryption for the purposes of hiding the data from a client is rather pointless... After all your client *must* know the key to the data, so all you have to do is find this - Encryption for this purpose is simply security by obscurity.

For those people using UDP, however, simple encryption can be a far more successful method of preventing spoofed packets causing the server to have your character do things you never intended it to do than TCP's simplistic sequence numbers - especially considering Windows' wonderfully lax implimentation.

In MMORPGs a combination of TCP streams and UDP based packet updates seems like a plan, or completely UDP based with a reliability algorithm of your own design (such things arn't *very* hard to do) for the bits you really do want to make sure arrive (I seem to recall the existence of the Reliable Datagram Protocol (RDP) RFC also, though I've not read it).
