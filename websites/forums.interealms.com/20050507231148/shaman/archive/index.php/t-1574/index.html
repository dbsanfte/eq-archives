Kieran
The real problem here is that the decision to use disease counters at all was completely flawed. And also equally inevitable. There's a big difference between changes that require code changes (particularly to a codebase that is probably in a fairly advanced state of decomposition by now) and those that are done by using the content tools.

The original aim was to make slows different from other effects that were dispelled using the "cancel magic" line, and to make them removable some other way. This could have been done in many, many ways.

The exact "right" way depends on internal implementation details of the spell system in the server. The "cure" spell effect could have been changed to recognise slows as curable directly, or else some new kind of effect could have been added like disease counters (as curses are separate) and coded to have no taunt effect. Whether a proper solution would have had any effect that we could see in the spdat is debatable since we don't have access to the server-side code.

Using the existing content tools and shoehorning things in to use the "disease counter" mechanism was a hack to avoid having to explicitly code anything to meet the new design goal - coders being a rare and expensive resource. Since all the existing spell effects have existing side-effects that are being relied upon, we have the domino effect of unforseen consequences that we've got to observe first hand.

Proper testing would have helped identify the fact that the "quick fixes" don't work, but then they would have had to face up to the cost of doing the job properly in the first place, which was what they were trying to avoid.
