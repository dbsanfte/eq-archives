    Re: Stigma Around Bittorrent

    I've worked with three IDE's - the first one, CodeWarrior, is supported specifically for education and was only used for my undergraduate work. It wasn't really all that great, but I also didn't know what to compare it against. The second one was at my first job - everyone had a choice of getting either Netbeans or UltraEdit as a code editor, and UE had a nice tie-in for the Atmel microcontroller I was working with. I still retain the lifetime license, today, but I don't use UE currently.

    The last IDE was used during my graduate studies with an AVR Dragon microcontroller. That IDE was fucking awesome and what every IDE should be. When you create a new project, a snazzy little window popped up (OMG) and would ASK you: do you want to enable the PWM? What freq? Great, done. Want some interrupts? How many? What priority (Drag-and-drop reordering)? Done. Oh, and I created the vectors for you, complete with "Insert-your-code-here" comments so you know WHERE to put your custom interrupt handler. And so on. Manually configuring all that crap is tedious and unenlightening. The manufacturer provided the IDE, so it covered every functionality of the microcontroller and got the student/developer to the Important Stuff quickly. That's how it should be.

    The stuff I use now isn't *really* an IDE. It's really more of a simulation environment (Synopsys VCS) with a bevy of Perl scripts surrounding it, maintained by a very large tools team. My code editing was done in a (very old) version of GEdit, since I had previously come from a Windows-based Ctrl-C Ctrl-V method of editing, until a coworker aliased it out. Seriously - a common environment include for all engineers had the line: alias 'gedit echo is deprecated. Please update to a modern code editor.'

    Asshole. I know who did it, too, and it was easy to work around.

    Nonetheless, I use GVIM now, if only because someone handed me a pretty snazzy .vimrc file and it supports the code folding he's put into many files. ./shrug Honestly, despite my preferences for UE over gvim, or Python over Perl, or ... it's all the same shit, and I work in ~5 different (very different!) code languages on a daily basis, anyway. I never really bother to keep track, just look at the file I'm editing and do more of the same thing. When I do refactoring work, I always get a reference out, anyway The only language I don't need one for, finally, is SystemVerilog, and that's only helpful to... verification engineers in silicon industry. Hm.

    The processor simulation stuff we have to support processor development is hardcore stuff. I actually develop the testbench for the fetch unit, for example, and it tells me the complete state & history of state of the entire block. However, this isn't something you can access commercially - it'd be like wanting to get a piston test harness from Ford for your own car engine work. It's doable, but of extremely limited use. But, per your far previous request, I can, in fact, see the complete contents of the processor at any given time.

    It's slow, though. About 10-20 cycles per second (CPS is what we call it, though a more common name would be Hertz) is awfully slow for a full core simulation. Even my single-unit bench only gets ~400 CPS, when all waveform debugging and dumping is disabled.

    Not exactly what you want to test your code with, I assure you.

    Re: Stigma Around Bittorrent

    And since no one asked, what I appreciate/found brilliant about vim, emacs, sam/acme from my investigations is

    With VIM
    1) Basing the text editing operations around a DSL that is extensible; that abstraction does not even have to be explicitly interacted with by the user, but enables user facing operations to exploit it. It is a very composable system. Acme's smart guessing could internally use this for instance to decide what expansions to make.

    2) VIM's modal based in combination with using key sequences instead of key chord (generally) greatly reduces the binding conflicts; the largely mnemonic grammar makes the solution space much smaller as well, and at least when learning the system conceptually instead of through rote memorization makes the learning curve much smaller. I have to explicitly memorize keybindings in other systems, usually through spaced repetition.

    Less positively:

    a) Using an esoteric, app specific programming language for a tool that targets a general audience is a terrible idea (note this is a different claim than making a DSL within a general purpose, non-app specific language)

    b) No matter how great the documentation for a program is, that is not going to stop the majority of users from not reading it while simultaneously complaining about something made evident had they read the documentation

    c) Unnecessarily not adopting ubiquitous UI standards elsewhere is a bad idea and will cause user aggravation, emacs also by default mimics that design decision bafflingly

    d) Simultaneously using an app specific language, then not explicitly making changes to it overtime evident in previously released code is a great way to make maintenance of the app difficult

    e) an unrefactored code base written in a variety of language specification changes is a great way to make maintenance of the app difficult and the cause of hair loss

    With Emacs
    1) While Vim's and Acme's design being to act as a tool with one specific purpose to use in combination with other system's tools (the UNIX philosophy) makes loads more sense abstractly, that dream has largely not been realized in practice sadly. Emacs internalizing tools overcomes that at the cost of becoming a kitchen sink, on the other hand this makes workflows more consistent across OS environments.

    2) Using the same language everywhere instead of making extensions writable in any language and not breaking feature sets consistently, makes learning a feature more worthwhile, having everyone write their plugins in the same language makes reading others code much easier as I do not have to be expert in ruby, lua, vimscript, c++, etc. to do so. The tool's culture largely benefits from this.

    3) Making an effort to encapsulate abstractions well is always worth doing so. That is pretty obvious but it is easy to get lazy and optimize for the short term.

    4) Having the source code available for all aspects of the system in question (in this case emacs) is tremendously beneficial to the point that not having such information is a tremendous handicap. Dtrace co-creator Bryan Cantrill made a similar point in one of his presentations at USENIX except in the context of the System Kernel in examination of Open Solaris and the then reclosed Solaris code. It is one of those things that was not readily apparent to me but once it clicks is enlightening. Why RMS had such a strong reaction to this early in life made sense afterwards.

    With Sam/Acme

    1) There is a huge room for improvement with how the mouse is utilized and it is yet another example of common sense being wrong that it is considered an inferior mode of interaction. Using the mouse as a source of chords to the fullest extent possible is one of those things that is obvious in retrospect but something I had not thought of until seeing it in action.

    2) Rob Pike is more genius than I will ever be.

    3) Rio's way of managing buffer windows etc. is really great for making a more efficient workflow and reduces cognitive load, which is not only nice for flow but removing tedium.

    4) Plumber is a really, really good idea and fantastically implemented.
