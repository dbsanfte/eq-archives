    Re: Ubuntu Unity

    I've been using emacs almost every day for about 5 years now and still don't really know how to use it. The on-the-fly macro recorder alone, though, puts it head and shoulders above anything else I've ever used. Most of the programming modes I use are fairly basic syntax highlighting/indenting. When I was writing a lot of C for a class, though I definitely had a whole bunch of templates that could easily be mapped to a single key (usually f8, for me). Like this one:

    Code:
    (defun ccc-node-predicate (fname)
  (interactive "sFunction Name:")
  (insert "int " fname "(ParseTreeNode *node){\n")
  (insert "  if (_OPERATOR==node->operator))\n")
  (insert "    return 1;\n")
  (insert "  else return 0;")
  (insert "}"))
    I'd use that template to generate code like this:
    Code:
    int isplus(ParseTreeNode *node){
  if (PLUS_OPERATOR==node->operator))
    return 1;
  else return 0;}

int isminus(ParseTreeNode *node){
  if (MINUS_OPERATOR==node->operator))
    return 1;
  else return 0;}
    This is a kind of boilerplate code that you get in C quite a bit, where the value of "DRY" is minimal (it's a boolean predicate function, that's it. If I write 30 of these, I'm never going to need to change all 30 of them at once, unless I change the "*_OPERATOR" name and no matter how compact I make it, there will still need to be 30 different tokens to change somewhere. So rather than write some potentially confusing error-prone preprocessor macro I just generate the code with emacs. I don't use it much in languages like python, though, because usually if you're generating code like this to save time you're probably doing something wrong.

    Regex replace is nice, too, though I find it hard to remember the syntax (having to escape every single paren is really annoying. I actually made a "baby's first youtube video" to give me practice and help me remember the syntax for the regex search: http://www.youtube.com/watch?v=0of7P-qKdkg

    I use a combination of eclipse and vim.

    Eclipse I use for C++ and PHP, primarily so I can just go to a function name, hit f3 and be brought to the function declaration (or list of functions). I used it when starting off to "figure out" the codebase. I have been told there exists something similar for vim, ctags.

    For smaller projects and my python work, I use vim exclusively.

    Biggest advantage for vim for me is being able to execute regular expressions between two marks. I'm sure everyone has had moments where they copy a block of code but change the variables to different names, for example. In eclipse, I need to use find/replace and stop at the end of the block. In vim I just ma at the top, mb at the bottom, then :'a,'b s/variable_a/variable_b/g .

    Re: Ubuntu Unity

    Alright, so I have been really working on my coding game lately and writing more complex models, and I am pretty much convinced a comparably powerful editor should be pushed and taught for anyone who writes a lot in tex or coding, especially coding. http://vimcasts.org/episodes/undo-br...and-gundo-vim/ Gundo is merely one of a
    myriad of slick features available in a highly developed editor. It's brilliant to use a tree based undo history over a linear design.

    VIM is stupefyingly esoteric and layered, however the keys and design for the various types are quite consistent, so I would say that while not intuitive at first it is intelligently designed with the power users in mind. VIM has many well ddeveloped plugins that all have support for all VIM mode types and are highly customizable. Registers, vim script, motions (combined with operators), folding, refactoring, mode based design, regex support, user profiles, UNIX tools (piping text in and out in various ways such as sed/awk/compiler messages) accessibility, incredibly well done documentation, marks, source cose management, and understanding of various tags is insane when utilized. You are the maestro and VIM is your symphony. Complex but beautiful.


    VIM tutor is great for novice introduction, but guiding people past that level within itself is where VIM's weakness lies. If someone put together an extension of the VIM tutor design with more lengthy material to work on it would greatly promote the VIM type editor paradigm instead of scaring people away. I even use it along side an IDE just to do the test editing, so you can keep your quick project creation, and visual debugging without having to hug a command line like it's 1968 all the time, which lets you keep GUI paradigms in your tool belt. The way I got better at exploiting my tips and tricks and work flows was by using VIM tutor, then watching screen casts, and googling for solutions to personal work flow needs/wants. The practical VIM book is supposed to be good too but I don't have first hand experience for review purposes.

    I'm not going to go out in the streets trying to convert 90s kids who live in GUIs but the alternative has its benefits too for the curious and persistant. I love the operator + motion concept so much, I wish more programs used that. Once it is engrained the process is like massaging Kate Upton: comfortable and sensational.
