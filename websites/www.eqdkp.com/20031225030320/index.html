The State of EQdkp (submitted by tsigo at 10:54pm EST)

Notice: The listed feature spec is being updated as I progress in coding v2.

It's been a long time since an update about EQdkp has been issued. Currently development is at a bit of a crossroad. Development on the 1.x branch has been slow, as I believe 1.2 fixed most of the issues with EQdkp.

But during the time since 1.2's release, I've been planning for the next major version of EQdkp. The changes have been so drastic that it will be known as EQdkp v2. It will support EverQuest as well as virtually all current and future games in which an attendance and item tracking system is plausible. Below is the current feature spec for EQdkp v2. Some pieces of it will only be interesting to developers, but users should be interested in most sections.

Please read through it and offer any thoughts at all in the forums. And if you have a "dream feature" that you haven't presented yet, now is the time to do so, while v2 is in the mostly-planning stage.

EQdkp v2 Feature Spec
---------------------

Modularity
----------
    Modules are bits of code (classes) that can be dynamically loaded, unloaded
    and re-ordered by EQdkp admins as they wish.  Each "break" in a page
    (i.e. each new table) could be a module.  For example, on the viewmember.php
    page: Raid Attendance is one module, Item History is another, Adjustment
    History is another. If the admin finds a module unnecessary he can hide it
    and its code will never be processed, increasing efficiency and decreasing
    load times.  Conversely, an admin could add new modules or rewrite existing
    modules to better suit his needs.  Even the order in which modules are
    displayed can be customized.

    Each module class is loaded by a base page - viewmember.php, for example -
    as needed/requested.  After being loaded, each module is responsible for
    running its code and assigning its template variables and nothing more;
    they are very specialized, and should eliminate confusing/clustered code
    where one page or class handles many different things.

Multi-game Capable
------------------
    Because of its modularity, EQdkp will support almost any MMO game.  All
    that's required is writing a new game module that tells EQdkp the game's
    classes, races, level range, log formats, etc.  Anything game-specific will
    be handled by a game class and not hard-coded in EQdkp.

    Because of this idea, the EQdkp name is a bit of a misnomer.  I'm thinking
    of keeping it because it already has a bit of recognition, but at the
    same time I'm open to suggestions.  "Meta Points" was considered at one
    point.

Template Engine
---------------
    Pages are now generated under the Smarty templating engine.  This allows
    for the use of custom plugins to make repetitive tasks much easier, and less
    variables will have to be assigned from PHP files.

    Note that because of JavaScript found within EQdkp templates, the Smarty
    left and right delimiters have been changed to "[" and "]", respectively.

    Custom plugins include:
        - block.lang.php
          [lang]LANGUAGE_INDEX[/lang]
          Grabs a language index from the $user-&gt;lang array, completely
          eliminating the language assignments from within PHP files.

        - function.bargraph.php
          [bargraph width="width" mod="1.0" color="(false|true|reverse)"]
          Creates a bargraph based on $width.  If width is not a percentage,
          it's multiplied by a modifier ($mod) to scale the bar.  If color
          is set to true, it uses the positive/negative/neutral colors to
          color the bar.  Reverse just means that 66 to 100% and 0 to 33% are
          reversed in their coloring.

        - function.color.php
          Replaces the color_item function from the EQdkp 1.x

        - function.glyph.php
          Displays a linked image glyph (or 16x16 icon) depending on the user's
          permissions.

        - function.link.php
          Displays a text link depending on the user's permissions.

        - function.phpself.php
          [phpself skip="variables,to,skip"]
          Returns a link to the current page, with all input variables (except
          those requested in $skip) appended to the URL as get variables.

        - function.pluralize.php
          [pluralize number="100" index="LANGUAGE_INDEX"]
          Depending on $number, would return "found 1 raid" or "found 100
          raids", for example.

        - function.sqlhighlight.php
          Highlights an SQL string

New Design
----------
    The default page layout of EQdkp has been slightly tweaked to showcase
    things like the user settings menus and login areas more prominantly. These
    new menus also eliminate the need for a separate admin area.  The admin
    links are available on every page if the user has the appropriate
    permissions.

Output Formats
--------------
    EQdkp recognizes different output formats and can be extended to use new
    ones. Currently supported are HTML and XML, but could output a page in
    comma-separated values (CSV) if needed.  XML could be used in the future to
    drive a desktop application allowing users to view the DKP standings outside
    their browser.

Internationalization
--------------------
    Locales are used instead of languages.  Locales include important data such
    as number formats which are now used in EQdkp, in addition to language
    settings. Numbers will show up as "1,329.00" instead of "1329.00" for the
    en_US locale.

    Language packs are also much smaller now.  Each module has its own language
    pack, so instead of loading one file with several hundred language
    definitions on every page, 1 to 10 files (depending on modules) are loaded
    with about a dozen definitions, resulting in less resource usage.

Completed Features
------------------
    These are features that work in their current state, but will of course be
    changed, improved and refined over time.

    Note that features already present in EQdkp 1.x will be listed here, as they
    are being rewritten for the modular foundation.

        - Module foundation
        - Dynamic menus
        - Style changing
        - Permission system; page checks for permissions and allows/blocks.
        - Very basic session foundation
        - Message system (output errors, instructions and general messages)

    Stats
    -----

    View Event

        - Attendance module (30, 60, 90 day, lifetime attendance percentages)

Planned Features
----------------
    These are features that are planned and/or on the wishlist.  Some may not
    make it in, some might not be possible, and more may be added.

    Note that these are in ADDITION to features that already exist in EQdkp 1.x.

    Global
    ------
        - [5] When a user is listing raids and is logged in, they'll be able to
          identify which raids their character is listed on and which they
          aren't. This is similar to icons in forums that show which topics
          you've posted in.
        - [9] Where checkboxes are present and it makes sense, have a "Select
          All" checkbox [#791653].

    Profiles
    --------
        - [4] Timezone offset.  Among other things, this will allow us to
          standardize times from raid logs [#806911].

    Items
    -----
        - [9] Item presets.  Populate your database with items and prices before
          they're dropped [#800818].
        - [2] On pages where an item purchase is shown, show the current or max
          value for that item to show inflation or padding.

    Raids
    -----
        - [6] Aftering entering attendees and choosing an event, etc., the admin
          will be able to change the value each member gets.  Very tentative.
        - [7] Parse a guild dump log (EverQuest module only).

    Configuration
    -------------
        - [9] There will be a separation between Anonymous permissions, and
          permissions automatically given to people when they become registered
          and authorized.
        - [8] Allow/disallow users to change usernames.
        - [6] Allow/disallow new registrations.
        - [2] Change images used as glyphs.
        - [3] Define which actions add logs.
        - [2] Define which actions send e-mails and where to.  Get an alert when
          an admin deletes a raid.
        - [5] Ban specific and ranges of IP addresses [#768252].
        - [5] Manage menus.  Re-order, enable/disable.
        - [9] When mass-updating user permissions, allow the admin to choose:
          Set permissions (set permission to these exactly), append permissions
          (add these new permissions to existing permissions), remove
          permissions (remove these permissions).
        - [6] Allow HTML in news posts [#786640].
        - [7] Allow SMTP mail server configuration.

    Logging
    -------
        - A user with appropriate permissions will be able to undo certain log
          actions. For example, if you're viewing an "Updated Raid" log, an
          admin will be able to revert the change made in that log.
        - Greater search capability.  Find logs where a certain member was
          removed from a raid, for example.
        - Archive/prune old logs [#802290].

    Miscellaneous
    -------------
        - [9] Prior to deleting a member, user, raid, possibly anything, add an
          option to export an SQL file containing all the data needed to restore
          that object [#815745].
        - [3] News generation.  Takes a date range and generates a news entry
          with "[Member] looted [Item][Screenshot] from [Raid]" type stuff.
        - [3] Admin notes.  Admins can leave notes to eachother (private to
          admins) or global notes (public to members who can see that page) on
          pages like viewmember to explain adjustments, viewraid to notify
          admins not to add a certain member, etc [#524842].
