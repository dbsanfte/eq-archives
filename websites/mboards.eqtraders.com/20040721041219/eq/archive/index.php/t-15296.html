Itek wrote:

"And having mapped that key to also be "move forward" ... Aim yourself into a corner... and wedge the "forward" key down with a coin... Presto ..."

This still won't work, because of how the keyboard works as a device. The control key is what the tech world calls a "state" key. The system sends a "turn on" code when the key is pressed, and a "turn off" code when it's released (all keys work this way, but only a few, like Ctrl, shift and Alt have an "on" and "off" state in the system). When you hold the key down, the signal for "Ctrl pressed" stays on, so when you click your mouse, you get control-click response from the EQ client. The problem is that other keys work differently (within the program; electrically, they do the same as the state keys). The up arrow isn't a "state" key, it's a "change" key. The program only reacts to the off-to-on or on-to-off change in state. If you press the key and hold it down, you run forward because the program moves you forward until it detects an on-to-off change, at which point it stops. The macro key reacts to the off-to-on change, and ignores the on-to-off change. Since pressing the movement key and wedging it down only gives you one off-to-on transition, the macro will fire once, and since the off-to-on never happens again, it'll never activate again, unless you let the key up and press it again. The only way around this is to use a keyboard-buffer stuffer that makes the system think you pressed a key when you didn't, and those programs are forbidden by the TOS agreement because they allow unattended macroing.

Sorry to say, but there is no easy way to forage without constant key presses. The same is true of begging, fishing and any other macro-keyed ability.
