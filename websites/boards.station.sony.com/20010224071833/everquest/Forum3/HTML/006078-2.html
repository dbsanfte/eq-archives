As stated earlier, the problem is with where the filtering is occuring. Turning off all those messages only filters it client-side, the data is still being sent. This is because its the easiest method to program, and since filters are generally for personal reasons, it's acceptable.

Now, for the server to filter this data instead of the client is unreasonable. Everyone has different settings, and the server would spend an enormous time logging who gets what data, it would be unreasonable.

HOWEVER, I propose this possible solution. One single command to tell the server to not send the unwanted set of data. To explain, the programmers get together to find out which data is the main culprit. Say it's spell info from other players, plus damage shield info from other players. Ok, so they build a server side filter, that blocks the sending of all hit/miss, spell, and dmg shield information that does not directly involve the PC himself. The activation command is /LowData (or whatever). While this command is toggled on, your client won't recieve any of that other info, whether you want it or not. No picking and choosing, cause that would tax the server too much trying to decide who gets what info. One single filter however should be feasible.

As it stands, I'm almost quitting this game over this issue. Every large encounter involves AE effects, it seems, and on nearly every large encounter I go LD. My poor little 33.6k connection cannot handle getting spammed with hit/miss, damage shield, and especially AE effect messages from 40 people. Therefore I go LD nearly every large encounter, and all I do is lag others with my presence. Very demoralizing, you know?

Just picking some arbitrary numbers to show what is happening:

Suppose that you get disconnected when you have 100 units of data backed up needing to be processed. This is when you go LD. Your client is getting too much info for it to handle, and it basicly throws up and says "I can't handle this any more. I'm shutting down". These units of data can be a damage hit from melee, a damage shield, a spell effect, etc. Let's also suppose that the client is able to process 40 units of data per time index.

Time index zero: Raid charges in.
Time index one:
+20 damage shield messages occur
+20 melee hits land
+20 spell effects happen from AE spell
-40 units of data are processed
=================
20 outstanding units of data left
Time index two:
+20 damage shield messages occur
+20 melee hits land
+20 spell effects happen from AE spell
-40 units of data are processed
=================
40 outstanding units of data left
Time index three:
+20 damage shield messages occur
+20 melee hits land
+20 spell effects happen from AE spell
-40 units of data are processed
=================
60 outstanding units of data left
Time index four:
+20 damage shield messages occur
+20 melee hits land
+20 spell effects happen from AE spell
-40 units of data are processed
=================
80 outstanding units of data left
Time index five:
+20 damage shield messages occur
+20 melee hits land
+20 spell effects happen from AE spell
-40 units of data are processed
=================
100 outstanding units of data left
MANY PLAYERS GOES LD
As players go LD the number of damage shield messages and melee hits and spell effects deminishes.
Time index five + (whatever time it takes for 1/3 the raid to go LD):
+13 damage shield messages occur
+13 melee hits land
+13 spell effects happen from AE spell
-40 units of data are processed
=================
-1 outstanding units of data left
(at this time, the client can finally start catching up on all of the previous time indexes it couldn't do. You are very lagged at this point, processing info from many time indexes beforehand.)

Basicly the ONLY way to fix this problem is to limit the amount of info being transmitted to the client. The current client is receiving all the damage shields and damage hits and so on, except that we can only choose not to display what has already been received. If we could choose not to receive it in the first place, then we wouldn't go LD, as we'd have a higher data unit processing rate than the incoming data.

Ester,

Verant advertises that you can play this game on a 28.8 or higher connection. Turning up your data rate sets the ceiling of the amount of throughout your connection can handle. You can set the data rate to 9.0 on a 56k modem and this may alleviate some of the problems, you still lag, and it is still unplayable.

Verant is missing the whole issue on link deaths. Link deaths occur, mostly on large or epic raids, due to the fact that the packets that are sent to the client are poorly constructed. Yes, that is the entire issue. Filtering data ONLY filters it from your client window. You STILL get all of the information and your connection gets congested. THIS is what causes link deaths on slow connections (33.6, 56k, ISDN).

The solution, and I am sure your programmers know this, is to redesign how packets are sent to the clients, what information is in them, etc. I can imagine that this has been passed by due to the fact that this would take a major re-write in code in order to accommodate this and that’s why we have not seen any changes. These issues can be fixed one of the following ways:

1) All of the filters need to be server side. The packets need to be built so that messages such as hits, misses, damage shields, spells, bard songs, etc. are dynamically added or removed from packets compared to how the character filter settings are set.

2) If dynamically building packets on the fly is to CPU intensive for the servers, then how about offering the users a RAID filter that allows Verant to have two methods of sending packets to clients. One is normal method, and the other is RAID. When RAID mode is set on, the client only gets the bare essentials needed to function. Filters are HARD CODED so that they don’t have to be dynamically built on the fly.

What Verant Fails to realize is that this reduction in packet size not only helps the user client, but it reduces the bandwidth needed to support the clients as well.

Just my two copper,
